# Splendor (Console / Java)

This repo contains our Splendor-like game project (Java).  
Before coding anything, please read this README fully — it explains the folder structure, how to compile/run, and our Git workflow.

---

## 0) Quick start (clone + run)

### Clone
```bash
git clone <REPO_URL>
cd <REPO_FOLDER>
```

### Compile + Run (required by assignment)
We use the required project structure:

```
compile.sh
run.sh
config.properties
src/        (all .java lives here)
classes/    (empty on submission; compiled .class goes here)
media/      (empty if console)
lib/        (optional jars)
```

Compile:
```bash
./compile.sh
```

Run:
```bash
./run.sh
```

> `compile.sh` compiles ALL Java files under `src/` into `classes/`.  
> `run.sh` runs `splendor.app.Main`.

## 1) Git rules (IMPORTANT)

### ✅ Never work directly on `main`
Always work on your own branch and submit a Pull Request (PR) when you want your changes merged.

### ✅ Branch naming convention (recommended)
Use a **hybrid** format so it’s clear **who owns the branch** and **what it’s for**:

```
<your-name>/<feature>
```

Examples:
- `danial/setup-scripts`
- `danial/card-loader`
- `alice/console-ui`
- `bob/game-engine-moves`

> Avoid name-only branches like `Danial` once you start multiple features — they get messy fast.

---

### Check your current branch
```bash
git branch
```

### Create a new branch (first time for a feature)
```bash
git switch -c <your-name>/<feature>
# example:
git switch -c danial/card-loader
```

### Switch to your branch (if it already exists)
```bash
git switch <your-name>/<feature>
# example:
git switch danial/card-loader
```


### ✅ Keep your branch up to date with `main`
Do this regularly before you start working (or before pushing a big change):

```bash
git switch main
git pull origin main
git switch <your-branch-name>
git merge main
```

If there are conflicts:
- fix them in VS Code
- then:
```bash
git add .
git commit -m "fix: merge main into my branch"
git push origin <your-branch-name>
```

### ❌ Never merge into `main` directly
If you want your changes merged into `main`, do this:

1. Push your branch:
```bash
git push origin <your-branch-name>
```

2. Open GitHub and create a **Pull Request**:
- base branch: `main`
- compare branch: `<your-branch-name>`

3. Wait for review/approval before merging.

## 2) Project package structure

All code lives under `src/` and uses the `splendor.*` package namespace:

```
src/
  splendor/
    app/       # entry point (Main.java)
    entities/  # data objects (Card, Noble, Player, GemBank, enums)
    data/      # loaders (CSV/properties readers)
    logic/     # game rules/engine (GameState, Move validation, turn logic)
    ui/        # console UI (rendering, input handling)
```

**Main entry point**:
- `src/splendor/app/Main.java`
- must contain:
  ```java
  package splendor.app;
  public class Main { public static void main(String[] args) { ... } }
  ```

---

## 3) Configuration (config.properties)

Winning condition is configurable via `config.properties` at the project root:

Example:
```properties
win.points=15
```

Your code should read this and default to 15 if missing/invalid.

---

## 4) Data files (custom deck)

We use **original (non-copyrighted) custom card/noble data** stored as CSV under:

```
src/data/cards.csv
src/data/nobles.csv
```

Recommended CSV schema:

**cards.csv**
```csv
tier,points,bonus,diamond,sapphire,emerald,ruby,onyx
1,0,DIAMOND,0,1,1,1,0
...
```

**nobles.csv**
```csv
id,points,diamond,sapphire,emerald,ruby,onyx
N1,3,4,4,0,0,0
...
```

---

## 5) Console UI output style (guideline)

We prioritize **usability over aesthetics**:
- show bank counts and current player status every turn
- refer to board cards by IDs (e.g., `T1-1`, `T2-3`, `N2`)
- keep output compact and easy to scan (no heavy ASCII art required)

---

## 6) Common gotchas

- Your folder structure MUST match your `package` lines.  
  Example:
  - file: `src/splendor/entities/GemColor.java`
  - package: `package splendor.entities;`

- `classes/` should be empty in submission (it will be generated by `compile.sh`).

## 7) Suggested workflow (small commits)

1. `git switch <your-branch>`
2. make a small change
3. `./compile.sh && ./run.sh` to sanity check
4. `git add .`
5. `git commit -m "<what you did>"`
6. `git push origin <your-branch>`
7. open a Pull Request (PR) if it should go into `main` (i.e., merge to `main` branch once feature is complete & stable)
